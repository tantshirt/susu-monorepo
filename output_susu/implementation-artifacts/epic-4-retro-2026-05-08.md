## What Went Well

- Strict stacked execution kept dependency order safe: 4.1 unlocked 4.2; 4.2 unlocked 4.3 and 4.5; 4.3 unlocked 4.4; 4.6 waited for the full payout surface.
- Auto-merge discipline worked: every PR merged only after GitHub CI and Cursor Bug Bot completed cleanly.
- Review gates caught substantive issues before merge, especially in 4.2, 4.4, 4.5, and 4.6.
- The implementation kept the no-scheduler invariant intact: payout and completion paths are permissionless but explicitly transaction-triggered.
- Test design before implementation gave each story a clear acceptance surface and made review easier.
- The final dependency graph now correctly shows Epic 5 readiness after Epic 4 completion.

## Challenges

- Story artifact drift appeared: stories 4.3, 4.4, and 4.5 still show `Status: ready-for-dev` in their individual files even though sprint status and dependency graph mark them done.
- Some acceptance criteria depended on a fuller live integration harness than the repo currently supports. The team had to use static/source-level checks plus Rust helper tests in places where a Surfpool or LiteSVM transaction harness would be stronger.
- The shared vault model introduced a non-obvious safety issue in 4.2: payout had to verify rotation funding so collateral could not be drained as pot funds.
- Guard ordering was easy to get wrong. In 4.4, funding verification initially ran before deadline rejection, which could have produced the wrong protocol error for early claims.
- Cursor Bug Bot found real test-quality problems in 4.5; the pipeline correctly blocked merge until those were fixed.
- `cargo fmt --check` has unrelated formatting drift on `main`, which creates noise for future quality gates.

## Key Lessons

1. Merge gates must include bot review, not just CI. Cursor Bug Bot produced actionable findings in 4.5.
2. Payout logic needs explicit economic-state validation. Token balance alone is not enough when collateral and contribution funds share a vault.
3. Guard order is protocol behavior. Error precedence must be part of the acceptance surface.
4. Static/source tests are useful, but they should not be mistaken for live transaction coverage.
5. BAD status artifacts need one authoritative update path. Drift across story files, sprint status, and dependency graph increases agent risk.
6. Stacked dependency execution is safer when each lower story is merged before descendants proceed.

## Significant Discoveries

No discovery invalidates Epic 5's direction, but three implementation realities must shape Epic 5:

1. Shared-vault semantics require precise economic proofs and tests.
   Impact: Epic 5 adversary and invariant work must model contribution funds and collateral distinctly.

2. Some Epic 4 claims are backed by static/source-level and Rust helper coverage rather than live Surfpool transaction traces.
   Impact: Epic 5 documentation must be careful about evidence strength and residual risk.

3. Artifact drift can mislead autonomous agents.
   Impact: Epic 5 should include an artifact consistency check before story selection.

Epic update required: no full Epic 5 rewrite required. Epic 5 execution should start with updated preflight assumptions and strict dependency graph use.
