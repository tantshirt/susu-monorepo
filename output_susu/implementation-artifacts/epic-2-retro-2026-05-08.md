## What went well

- **Incremental program delivery:** Instructions landed in dependency order (accounts → create → invite → accept → cancel), reducing rework.
- **Test harness maturity:** Program tests (invite, accept, cancel, happy path) gave fast feedback and caught regressions early.
- **CI as contract:** IDL hash, forbidden patterns, SDK parity, Anchor build, and workspace tests created a shared definition of green.
- **Story 2.6:** Explicit SDK helpers + Vitest mocks; Rust `susu-client` using canonical `susu` program types for deserialize keeps alignment with on-chain layout.
- **Automation discipline:** BAD-style pipelines (worktrees, PRs, merge gates) kept story 2.6 reviewable and auditable.

---

## Challenges (systems, not blame)

- **Codama / codegen expectations:** Generated TS/Rust account types were often **stub** `{ raw }` shells; helpers had to bridge with layout-aware parsing or Anchor deserialize in Rust. Risk: integrators expect rich typed decoders before codegen catches up.
- **Tracker vs GitHub:** Local `sprint-status` / graph rows can lag merges; Phase-0-style reconciliation must stay routine.
- **External checks:** Cursor Bugbot / Autofix timing vs merge — clarify which checks are required before merge to avoid ambiguity.
- **Lockfiles:** Root `Cargo.lock` generation is inconsistent with "no lockfile in repo" bootstrap; policy should be explicit (commit vs generate in CI only).

---

## Lessons learned

1. **Program-first, SDK-follow:** Implementing on-chain state before query helpers avoided designing reads against a moving schema.
2. **Parity is a process:** TS mocks + Rust `susu` types can both be "green" while Codama still emits placeholders — document the gap until codegen matures.
3. **memcmp and layouts:** Participation history filtering depends on stable `MemberPosition` field order; comments and shared constants reduce silent drift.
4. **Merge gates:** Waiting for CI (and agreed bot checks) before squash-merge reduced broken `main` risk.
